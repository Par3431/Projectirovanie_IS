# Нефункциональные требования (магазин автозапчастей)

## 1. Адаптивный дизайн (Responsive UI)

### Ширина экрана
- от 320px до 1920px — нет горизонтальной прокрутки

### Таблицы с артикулами и ценами
- на мобильном устройстве таблица превращается в «карточки товара» (фото, артикул, цена, кнопка «В корзину»)
- без горизонтального скролла

### Фильтры (бренд, цена, совместимость)
- на десктопе — сайдбар слева
- на мобильном устройстве — выезжающая шторка снизу (drawer) по кнопке «Фильтры»

### Кнопки
- зона клика не менее 44×44 pt (по Apple HIG)
- на мобилке кнопка «В корзину» прилипает к низу экрана при скролле

## 2. Наличие полноценной веб-версии (Web application, не PWA-lite)

### Конкретные требования

**SPA или MPA с прогрессивным рендерингом:**
- перезагрузка страницы не должна сбрасывать корзину и поиск
- при обновлении страницы — восстановление состояния (последний поиск, выбранные фильтры)

**Работа офлайн (частично):**
- Service Worker кэширует: статику (CSS, JS), список последних 10 поисков по VIN, корзину
- при потере сети — показывать «Вы офлайн, заказ будет отправлен при восстановлении соединения»


## 3. Защита персональных данных (соответствие 152-ФЗ для РФ)

### Согласие на обработку ПДн
- отдельный чекбокс при регистрации/оформлении заказа (не встроенный в оферту)
- возможность отозвать согласие через личный кабинет
- логирование факта выдачи согласия (время, IP, User-Agent)

### Минимизация данных
- не хранить паспортные данные (если не требуется для таможни)
- VIN хранить в отдельной таблице с ограниченным доступом

### Доступ по ролям
- менеджер склада видит только адрес и телефон (номер заказа)
- аналитик видит обезличенные данные (город, категория авто, но не ФИО)
- администратор поддержки — полные данные, но все действия логируются

### Право на забвение
- при запросе пользователя удалить все его данные (кроме тех, что обязаны хранить 5 лет по налоговой) — полное удаление из всех БД и бэкапов

### Хранение
- серверы только на территории РФ (Yandex Cloud / Selectel / свой ЦОД)
- шифрование на диске (LUKS или AES-256 для томов БД)


## 4. Хеширование паролей пользователей (Password storage security)

### Алгоритм хеширования
- только **bcrypt** с cost factor = 12 (или 14 для критичных аккаунтов администраторов)
- запрещены: MD5, SHA-1, SHA-256 без соли, собственные алгоритмы

### Соль (salt)
- генерируется отдельно для каждого пароля (автоматически bcrypt)
- дополнительная site-wide соль в конфигурации приложения (секрет)

### Политика паролей (на уровне требований к проекту)
- минимальная длина 8 символов
- хотя бы одна цифра и одна заглавная буква (можно на клиенте и сервере валидировать)

### Защита от брутфорса входа
- после 5 неудачных попыток — CAPTCHA
- после 10 — временная блокировка на 15 минут (по login+IP)

### Хранение в БД
- поле `password_hash` тип VARCHAR(255)
- индекс для логина/email, но не для хеша пароля

### Обязательное требование к проекту
- ни в коде, ни в логах, ни в отладке не должно быть вывода `$password` или `plain_text_password`


## 5. Документация API (для интеграции с поставщиками и фронтом)

### Формат документации
- **OpenAPI 3.0** (Swagger) в файле `openapi.yaml` или `swagger.json`
- документация всегда доступна по URL: `/api/docs/` и `/api/docs/swagger-ui` (интерактивная)

### Какие эндпоинты обязательно описать

| Метод | Эндпоинт | Назначение |
|-------|----------|-------------|
| GET | `/api/v1/search?vin=...` | поиск по VIN |
| GET | `/api/v1/stock/{article}` | получение остатков |
| POST | `/api/v1/orders` | создание заказа |
| GET | `/api/v1/orders/{id}` | проверка статуса заказа |
| POST | `/api/v1/auth/login` | авторизация / регистрация (JWT) |

### Содержание каждого эндпоинта
- метод, URL, заголовки (Content-Type, Authorization)
- параметры запроса (query, path, body) с типами, обязательностью, примером
- возможные HTTP статусы (200, 400, 401, 403, 404, 500) и примеры ответов

### Аутентификация в API
- описание механизма JWT Bearer Token
- пример получения токена (POST `/api/v1/auth/login`)

### Версионирование
- префикс `/api/v1/` — для текущей версии
- при изменениях — создавать `/api/v2/` без поломки старых клиентов

### Для внешних поставщиков (B2B API)
- отдельный раздел документации с ограниченной квотой (rate limits)
- примеры запросов на curl / Python / JS
