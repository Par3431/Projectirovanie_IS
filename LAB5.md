# Функциональные требования (магазин автозапчастей)

## 1. Хранение полной информации о товарах

### Состав атрибутов товара

- **Артикул** (уникальный строковый ID, обязательный)
- **Наименование** (строка, до 255 символов)
- **Бренд** (справочник, выбор из существующих или ввод нового)
- **Категория** (иерархический справочник: «Тормозная система → Колодки»)
- **Вес** (кг, с точностью до 0.01)
- **Габариты** (Д×Ш×В в см, три поля)
- **Штрихкод** (EAN-13, уникальный, опционально)
- **Изображения** (до 5 шт., форматы JPEG/PNG/WEBP, не более 5 МБ каждое)
- **Совместимость** (связь с таблицей моделей авто: марка, модель, год, VIN-маска)
- **Остаток на складе** (целое неотрицательное число)
- **Цена закупки** (рубли, копейки, decimal)
- **Цена продажи** (рубли, копейки, >= цена закупки)
- **Место хранения** (стеллаж/ряд/полка, строковое поле)

### Технические ограничения

- Все текстовые поля должны поддерживать поиск с учетом морфологии русского языка
- Изображения хранятся в отдельном хранилище (S3 или файловая система), в БД — ссылка
- При удалении товара — помечать как `deleted`, физически не удалять (для истории продаж)

## 2. Хранение данных о клиентах (физлица/юрлица)

### Общие атрибуты (для обоих типов)

- **Тип клиента**: `individual` / `legal`
- **Контактный телефон** (обязательный, используется как логин)
- **Email** (опционально)
- **История покупок** (ссылки на заказы, автоматически)

### Для физлица

- **ФИО** (три отдельных поля или одно)
- **Адрес доставки** (строка)
- **Список автомобилей клиента** (марка, модель, год, VIN) — до 5 авто

### Для юрлица

- **Название организации**
- **ИНН** (уникальный, валидация по контрольной сумме)
- **КПП** (опционально)
- **Юридический и фактический адрес**
- **Контактное лицо** (ФИО + должность)
- **Номер и дата договора**
- **Условия отсрочки платежа** (количество дней, максимальная сумма долга)

### Требования к интерфейсу

- При вводе телефона — автоподстановка клиента (если существует)
- При смене статуса на `legal` — открываются дополнительные поля для ИНН и договора

## 3. Ведение справочника поставщиков

### Атрибуты поставщика

- **Название** (уникальное)
- **Контактные данные**: телефон, email, контактное лицо
- **Условия оплаты**: предоплата / отсрочка X дней / постоплата
- **Типовой срок поставки** (в днях, целое число)
- **Рейтинг надежности** (вычисляемое поле: 1–5 звезд)
- **История заказов этому поставщику** (ссылочно)

### Автоматическое обновление рейтинга

- Система снижает рейтинг, если поставщик:
  - задержал поставку >20% от срока → **–0.5**
  - поставил брак (отмечено при приемке) → **–1.0**
- Повышает рейтинг при досрочной поставке (>3 раз подряд) → **+0.2**

### UI требование

- При выборе товара для заказа — система показывает таблицу:  
  `поставщик → цена → срок → рейтинг → рекомендованный` (выделен зеленым)

## 4. Формирование заказов поставщикам

### Типы заказов

- **Плановый** — автоматически при снижении остатка ниже `min_remnant`
- **Срочный** — вручную менеджером под конкретный клиентский заказ

### Алгоритм формирования планового заказа

1. Система сканирует товары, где `current_stock <= min_remnant`
2. Формирует предложение: товар → рекомендованный поставщик → количество до `max_stock`
3. Менеджер корректирует и подтверждает

### Выходные данные заказа

- Номер заказа, дата, список позиций (артикул, количество, цена закупки)
- Статусы: `Черновик` → `Отправлен поставщику` → `В пути` → `Принят частично` → `Принят полностью`
- Сгенерированный PDF-файл счета/заказа (шаблон настраивается)

### Интеграция

- Опционально: экспорт заказа в XML/JSON по API поставщика (если поддерживает)

## 5. Поиск товара в базе сервиса

### Поддерживаемые типы поиска

| Тип поиска | Пример |
|------------|--------|
| По артикулу | `R1411036` |
| По штрихкоду | сканером |
| По наименованию | «колодки тормозные тойота» |
| По VIN-коду | `JT153S5000000001` |
| По марке/модели/году | Toyota Camry 2018 → «фильтр масляный» |

### Результат поиска (отображается списком карточек)

- Фото, артикул, наименование, бренд
- Наличие на складе (зеленый/красный индикатор)
- Цена продажи
- Если нет в наличии: «Под заказ X дней» и кнопка «Уведомить»

### Особые требования

- При поиске по VIN — автоматически показать кнопку «Добавить этот авто в мои» (для авторизованных)
- Поиск должен работать без перезагрузки страницы (AJAX)

## 6. Формирование отчетов

### Доступные типы отчетов и их содержание

| Тип отчета | Поля/показатели |
|------------|----------------|
| Финансовый (прибыль) | Выручка, себестоимость, валовая прибыль, маржинальность |
| Дебиторская задолженность | Клиент (юрлицо), сумма долга, просрочка дней |
| Складские остатки | Артикул, наименование, остаток, место хранения |
| Оборотная ведомость | Приход, расход, остаток на начало/конец периода |
| Неликвиды | Товары без продаж >90 дней |
| Продажи по менеджерам | Менеджер, кол-во заказов, сумма, средний чек |
| Топ продаж | Топ-20 товаров по штукам, сумме, маржинальности |

### Форматы выгрузки

- **Excel** (.xlsx) с форматированием (жирный заголовок, автоширина)
- **PDF** с шапкой организации и подписью
- **CSV** (для аналитиков)

### UI требование

- Кнопка «Сформировать» открывает окно предпросмотра перед скачиванием

## 7. Регистрация и авторизация пользователей

### Роли и права (матрица доступа)

| Действие | Админ | Менеджер | Кладовщик | Бухгалтер | Кассир |
|----------|-------|----------|-----------|-----------|--------|
| Поиск товара | ✅ | ✅ | ✅ | ❌ | ✅ |
| Создание заказа клиента | ✅ | ✅ | ❌ | ❌ | ✅ |
| Приемка товара | ✅ | ❌ | ✅ | ❌ | ❌ |
| Работа с отчетами | ✅ | частично | ❌ | ✅ | ❌ |
| Управление пользователями | ✅ | ❌ | ❌ | ❌ | ❌ |

### Регистрация нового пользователя (только для администратора)

1. Админ заполняет: email, роль, ФИО
2. Система отправляет на email ссылку вида `/set-password?token=...` (срок жизни 48 часов)
3. Пользователь устанавливает пароль
4. Первый вход — обязательная смена временного пароля

### Авторизация

- Форма: телефон/email + пароль
- Успешный вход → выдача JWT-токена (access 8ч, refresh 7 дней)
- Неудачные попытки: счетчик логируется, блокировка по сценарию из нефункциональных требований

## 8. Формирование и обработка заказов клиентов

### Статусная модель заказа (машина состояний)
Корзина → Оформлен → Оплачен → В сборке → Собран → Выдан
↓ ↓
Отменен Отменен

text

### Действия при каждом статусе

| Статус | Кто выполняет | Действие системы |
|--------|---------------|------------------|
| Корзина | Клиент/менеджер | Добавление товаров, изменение количества |
| Оформлен | Менеджер | Блокировка товара (резерв на 30 минут), запрос оплаты |
| Оплачен | Кассир / система | Снятие резерва, списание остатков, отправка задачи кладовщику |
| В сборке | Кладовщик | Отображение списка товаров с местом хранения |
| Собран | Кладовщик | Отметка о готовности, уведомление клиента |
| Выдан | Кассир/менеджер | Закрытие заказа, печать чека и гарантии |
| Отменен | Система/менеджер | Возврат товара на склад (если был резерв или оплата) |

### Обязательные поля заказа

- **Номер заказа** (формат: `ORD-20250608-0001`)
- **Дата и время создания**
- **Список позиций** (артикул, количество, цена на момент продажи)
- **Итоговая сумма**
- **Способ оплаты** (наличные / карта / безнал)
- **Способ получения** (самовывоз / доставка)

## 9. Ведение статистики продаж

### Автоматически собираемые метрики

| Метрика | Где отображается |
|---------|------------------|
| Продажи сегодня / вчера / за месяц | Дашборд на главной |
| Количество заказов за сегодня | Дашборд |
| Средний чек по магазину | Дашборд |
| Конверсия (оформлен → оплачен) | Отчет для менеджеров |
| Топ-5 брендов по продажам за неделю | Блок на главной |
| Топ-5 категорий | Блок на главной |
| Среднее время от оплаты до выдачи | KPI склада |

### Требование к визуализации

- Использовать графики (Chart.js / ECharts): линейный график продаж по дням, круговая диаграмма по брендам
- Данные обновляются без перезагрузки страницы (WebSocket или polling каждые 60 секунд)

## 10. Импорт и экспорт данных

### Импорт (загрузка в систему)

| Тип данных | Источник | Формат | Алгоритм |
|------------|----------|--------|----------|
| Товары + остатки | Файл от поставщика | XLSX, CSV | По артикулу: если есть — обновить, нет — создать |
| Номенклатура | Выгрузка из 1С | XML (CommerceML) | Полная синхронизация |
| Клиенты (оптовые) | Загрузка из CRM | CSV | Сопоставление по ИНН или телефону |

### Экспорт (выгрузка из системы)

| Тип данных | Получатель | Формат | Фильтры |
|------------|-----------|--------|---------|
| Заказы за период | Бухгалтерия (1С) | XML, JSON | По дате, по статусу |
| Складские остатки | Маркетплейс | CSV, YML | По брендам/категориям |
| Финансовый отчет | Директор / налоговая | PDF, Excel | За период |

### Требования к процессу импорта

- Перед импортом — предпросмотр: первые 5 строк и количество записей
- При ошибке в строке — пропустить её, залогировать, продолжить
- Отчет об импорте: «Загружено X, обновлено Y, ошибок Z»
