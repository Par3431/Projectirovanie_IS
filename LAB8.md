# Архитектура интернет-магазина автозапчастей

## Раздел 1 — Выбор архитектурного стиля

**Выбранный стиль:** Клиент–сервер (с монолитным бэкендом)

### Верхнеуровневая схема
[Браузер] ———— REST API/HTTPS ———— [Веб-сервер + Монолитное приложение] ———— [PostgreSQL]
| |
| +———— [Redis (кеш)]
[Моб. приложение] ———— REST API/HTTPS ——————————+

text

**Клиенты:** браузер (SPA) и мобильное приложение (iOS/Android)  
**Сервер:** один монолитный бэкенд  
**Протокол:** HTTPS + JSON/REST

### Обоснование

#### 1. Почему этот стиль подходит для задачи?

Магазин автозапчастей — это стандартный CRUD с каталогом, поиском и оформлением заказа. Бизнес-логика централизована, нет сложных распределённых сценариев. Один бэкенд одинаково обслуживает веб и мобилку через единый API. Монолит внутри позволяет быстро разрабатывать и деплоить фичи (например, новый фильтр запчастей).

#### 2. Почему отказались от альтернатив?

Микросервисы — избыточны. У нас одна команда из 3–5 разработчиков, нагрузка небольшая (сотни DAU). Разбивка на «Каталог», «Корзину», «Заказы» принесёт сложности с транзакциями (оформление заказа требует атомарного списания остатков) и DevOps-оверхед. Отказ от микросервисов — чтобы не лечить «распределённую боль».

#### 3. Минусы выбора и как жить

- **Монолит растёт** → усложняется код → применяем чёткую модульную структуру (раздел 5) и выделяем слои.
- **Масштабирование** — только вертикальное или копированием монолита → при росте нагрузки ставим несколько копий за балансировщиком.
- **Единый стек технологий** → не страшно, т.к. нет подзадач, требующих разных языков.

### Условие перехода с монолита (на клиент-сервер с микросервисами или просто на микросервисы)

Когда:
- DAU вырастет до 10 000+ **и**
- появятся чёткие команды (например, 5+ человек на каталог, 5+ на заказы) **и**
- потребуется независимый деплой (например, обновляем систему скидок без остановки оформления заказов)

### Компоненты

| Компонент | Что делает | Зачем в моей системе |
|-----------|------------|----------------------|
| Браузер | Отображает каталог, корзину, оформление заказа | Веб-интерфейс для пользователей с ПК/ноутбуков |
| Моб. приложение | То же самое + push-уведомления об изменении статуса заказа | Автовладельцы часто ищут запчасти с телефона (в гараже, на дороге) |
| Nginx | Принимает HTTPS, отдаёт статику (фронтенд), проксирует API на бэкенд, простой балансировщик | Чтобы один сервер мог обслуживать и статику, и API + позже легко добавить второй инстанс бэкенда |
| Монолит бэкенд | Каталог, поиск, корзина, заказы, расчёт доставки, аутентификация | Вся логика магазина в одном месте — просто транзакции по заказу |
| PostgreSQL | Хранит пользователей, запчасти (категории, артикулы, цены, остатки), заказы, корзины | Данные реляционные (связи много ко многим), нужны транзакции при резервировании товара |
| Redis | Кеширует результаты поиска по запчастям (по популярным запросам: «тормозные колодки Bosch») и сессии пользователей | Поиск в каталоге автозапчастей тяжёлый (LIKE по артикулам, фильтрация). Без кеша при 500 DAU БД будет грузиться одинаковыми запросами каждые 5 секунд |

### Почему нет брокера сообщений

Все операции синхронны. Пользователь не отправляет запрос на «расчёт доставки через час» — ему нужно сразу видеть стоимость при оформлении. Push-уведомления отправляются синхронно через Firebase/APNS, не нужна очередь.

## Раздел 3 — Оценка нагрузки

### 1. DAU

Допустим, магазин автозапчастей средней руки.  
**DAU = 500 человек** (реалистично: нишевой магазин, автолюбители заходят проверить цену/наличие)

### 2. Действия одного пользователя в день

- Просмотр страниц каталога: 10 раз
- Поиск запчасти (по артикулу или названию): 3 раза
- Добавление в корзину: 1 раз
- Оформление заказа: 0,2 раза (раз в 5 дней)

### 3. RPS (requests per second)

**Среднее количество действий в день от DAU:**

- Просмотр каталога: 500 × 10 = 5 000
- Поиск: 500 × 3 = 1 500
- Добавление в корзину: 500 × 1 = 500
- Оформление заказа: 500 × 0,2 = 100

**Всего действий в день = 7 100**

Дневная нагрузка распределена не равномерно. Основные часы (10:00–22:00) — 12 часов.  
Секунд в 12 часах = 12 × 3600 = 43 200 с.

- **Средний RPS** = 7 100 / 43 200 ≈ 0,16 RPS
- **Пиковый RPS** (× 3 от среднего) = 0,48 RPS

А если условимся, что в час-пик (вечером с 19 до 20) совершается 20% дневных действий:  
20% от 7 100 = 1 420 действий в час → 1 420 / 3600 ≈ **0,4 RPS** в час-пик.

**Вывод:** реальная нагрузка — меньше 1 RPS даже в пик.

### 4. Данные

- Одна запись о заказе: ~2 КБ
- Заказов в день: 100 (0,2 × 500)
- Объём заказов в день = 100 × 2 КБ = 200 КБ

Пользователи, корзины, запчасти — пишется редко. Основной объём — каталог (запчасти) статичен.  
В день приращение = ~0,2 МБ (только заказы и логи).  
За год = 0,2 МБ × 365 ≈ 73 МБ — ничтожно.

### 5. Вывод по масштабированию

Одного сервера (2 CPU, 4GB RAM) достаточно на годы вперёд.  
Горизонтальное масштабирование не нужно. Redis и PostgreSQL уживаются на той же машине. Единственная причина добавить второй сервер — отказоустойчивость (но по заданию это не требуется).

## Раздел 4 — Выбор стека технологий

| Слой | Выбор | Обоснование | Альтернатива и отказ |
|------|-------|-------------|----------------------|
| Бэкенд | Python + FastAPI | Высокая скорость разработки, лёгкий async для параллельных запросов к PostgreSQL и Redis. Хорошо работает с Pydantic для валидации запросов к API. | Django (тяжёлый, синхронный). FastAPI легче под микросервисную подготовку в будущем. |
| База данных | PostgreSQL | Нужны транзакции (остатки товара при оформлении), сложные запросы (поиск по артикулам, JOIN каталог-цены-остатки). Атомарность ключевая. | MongoDB (не поддерживает транзакции на коллекциях — риск конфликтов остатков) |
| Фронтенд (веб) | React + TypeScript | Интерактивный каталог, корзина без перезагрузки страницы. TypeScript ловит ошибки типов данных API (цены, артикулы). | Vue (тоже норм, но у команды больше опыта в React) |
| Кеш | Redis in-memory | Кеш результатов поиска по запчастям (TTL = 10 мин) и сессий. Дешевле, чем Memcached, но с типами данных (словарь результатов). | Memcached (только ключ-значение, неудобно для сложного кеша) |

## Раздел 5 — Архитектура кода (бэкенд)

### Слои и их взаимодействие
HTTP-запрос (Nginx)
↓
[Controller / Router] → валидация тела запроса (Pydantic)
↓
[Service layer] → бизнес-логика
↓
[Repository / DAL] → SQL-запросы
↓
PostgreSQL / Redis

text

### Конкретные модули магазина автозапчастей

| Модуль | Controller | Service | Repository |
|--------|------------|---------|------------|
| catalog | `GET /parts?category=brakes` | Фильтрация, пагинация | `SELECT * FROM parts WHERE ...` |
| search | `GET /search?q=bosch+колодки` | Вызов Redis (если есть кеш), иначе — поисковый запрос | `ILIKE` по артикулу и названию |
| cart | `POST /cart/add` | Проверка остатка, пересчёт цены | `UPDATE cart_items` |
| orders | `POST /orders` | Резервирование остатков (транзакция), создание заказа | `BEGIN; INSERT orders; UPDATE stock; COMMIT;` |
| auth | `POST /login` | Проверка пароля (bcrypt), создание JWT | `SELECT FROM users WHERE email=?` |

### Схема (текстовое описание)
[API Router]
├─ auth_controller → auth_service → user_repo (User table)
├─ catalog_controller → catalog_service → part_repo (Parts)
├─ search_controller → search_service → search_repo (SQL + Redis)
├─ cart_controller → cart_service → cart_repo (Redis сессии или БД)
└─ order_controller → order_service → order_repo + stock_repo (транзакция в БД)
