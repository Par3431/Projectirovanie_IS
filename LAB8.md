# Архитектурное проектирование: интернет-магазин автозапчастей

## Раздел 1 — Выбор архитектурного стиля

### Выбранный стиль: Клиент–сервер (монолитный бэкенд)

### Верхнеуровневая схема

```mermaid
graph LR
    A[Браузер] -->|HTTPS + REST API| C[Веб-сервер + Монолит]
    B[Мобильное приложение] -->|HTTPS + REST API| C
    C -->|SQL| D[PostgreSQL]
    C -->|Кеш| E[Redis]
Клиенты: браузер (SPA на React) + мобильное приложение (Flutter/Kotlin)
Сервер: монолитное приложение на Python/FastAPI
Протокол: HTTPS, JSON, REST API

Обоснование выбора
Почему этот стиль подходит?
Магазин автозапчастей — классическая e-commerce система с каталогом, поиском, корзиной и заказами. Все операции транзакционны и требуют атомарности (оформление заказа = резервирование остатков). Монолит позволяет:

Держать всю бизнес-логику в одном месте

Использовать простые ACID-транзакции без распределённых саг

Быстро разрабатывать новые фичи (одна команда из 3–5 человек)

Почему отказались от альтернатив?
Микросервисы — отвергнуты по следующим причинам:

Избыточны при нагрузке < 1000 DAU

Потребовали бы распределённых транзакций (оформление заказа = атомарность по каталогу, корзине, остаткам)

Команда слишком мала для DevOps-нагрузки (Kubernetes, service discovery, circuit breakers)

Нет потребности в независимом деплое разных сервисов

Минусы выбора и как с ними жить
Минус	Решение
Монолит разрастается до "Big Ball of Mud"	Чёткая слоистая архитектура (см. Раздел 5) + модульность по доменам
Единый стек технологий	Нам и не нужно разных языков — вся логика на Python
Только вертикальное масштабирование	При росте нагрузки > 2000 DAU ставим несколько копий за балансировщиком
Один баг — падение всего сервиса	Качественное тестирование, CI/CD, мониторинг, healthchecks
Условия перехода на микросервисы
Переход произойдёт при выполнении двух или более условий:

DAU вырастет до 10 000+

Появятся отдельные команды разработки (5+ человек на каталог, 5+ на заказы)

Потребуется независимый деплой (например, обновление системы скидок без остановки оформления заказов)

Появится интеграция со сторонними сервисами доставки с разными протоколами

Описание компонентов
Компонент	Что делает	Почему нужен именно в моей системе
Браузер	Отображает каталог, корзину, оформление	ПК-пользователи ищут запчасти дома/в офисе
Моб. приложение	Тот же функционал + push-уведомления	Автовладельцы ищут запчасти прямо в гараже или на дороге через телефон
Nginx	Отдаёт статику, проксирует API на бэкенд, балансирует	Один сервер обслуживает и фронтенд, и API. Позволит позже добавить второй инстанс бэкенда без изменения клиентов
Монолит (FastAPI)	Вся бизнес-логика: каталог, поиск, корзина, заказы, аутентификация	Простота транзакций: резервирование остатков при заказе требует одного SQL BEGIN/COMMIT
PostgreSQL	Хранит пользователей, запчасти, остатки, заказы	Данные строго реляционные (артикулы → категории → цены → остатки). Нужны JOIN и транзакции
Redis	Кеширует результаты поиска (TTL 10 мин) и пользовательские сессии	Поиск по автозапчастям — тяжёлый: ILIKE по артикулу + фильтрация по 5 полям. Без кеша при 500 DAU один и тот же запрос («колодки Bosch») бил бы БД каждые 5 секунд
Почему нет лишних компонентов
Брокер сообщений (RabbitMQ/Kafka) — не нужен. Все операции синхронны:

Пользователь не отправляет запрос на "расчёт доставки через час"

Push-уведомления отправляются напрямую через Firebase/APNS

Нет фоновой обработки изображений или долгих отчётов

CDN — не нужен при текущей нагрузке и объёме статики (картинки запчастей маленькие, подгружаются из Nginx).

Раздел 3 — Оценка нагрузки (Back-of-the-envelope)
Исходные допущения
Средний нишевый магазин автозапчастей

DAU = 500 человек

Рабочие часы: 12 часов (10:00 – 22:00)

Действия одного пользователя в день
Действие	Количество	Пояснение
Просмотр страниц каталога	10	Изучает категории (тормозные диски, колодки и т.д.)
Поиск запчасти	3	Ищет по артикулу или названию
Добавление в корзину	1	—
Оформление заказа	0,2	Раз в 5 дней
Расчёт RPS
Всего действий в день:

text
500 × 10 = 5 000 (просмотры)
500 × 3 = 1 500 (поиск)
500 × 1 = 500 (добавление в корзину)
500 × 0,2 = 100 (заказы)

Итого = 7 100 действий/день
Часовой пик (с 19:00 до 20:00) — 20% дневной нагрузки:

text
7 100 × 0,2 = 1 420 действий/час
1 420 / 3600 ≈ 0,39 RPS в час-пик
Пиковый RPS с запасом (×3):

text
0,39 × 3 ≈ 1,17 RPS
Вывод: максимальная нагрузка — ≈ 1 запрос в секунду в пик.

Хранимые данные
Тип данных	Размер	Прирост в день	За год
Заказ (~1 КБ)	1 КБ	100 заказов × 1 КБ = 100 КБ	≈ 36 МБ
Пользователь (~2 КБ)	2 КБ	10 новых × 2 КБ = 20 КБ	≈ 7 МБ
Логи действий (~0,5 КБ)	0,5 КБ	7 100 × 0,5 КБ ≈ 3,5 МБ	≈ 1,3 ГБ
Итого	—	≈ 3,7 МБ/день	≈ 1,35 ГБ/год
Каталог запчастей (~10 000 позиций) — статичен, грузится один раз.

Вывод: нужен ли горизонтальный масштабирование?
Сейчас — нет. Один сервер с 2 vCPU и 4 GB RAM легко справится с 1 RPS.

Достаточно:

Одна виртуальная машина

PostgreSQL + Redis на той же машине

Nginx в качестве прокси

Горизонтальное масштабирование потребуется при:

DAU > 3000 (нагрузка ≈ 6–7 RPS)

Или требовании отказоустойчивости (тогда ставим второй инстанс бэкенда + реплику БД)

Раздел 4 — Выбор стека технологий
Слой	Выбор	Обоснование	Альтернатива (почему нет)
Бэкенд	Python 3.12 + FastAPI	— Async из коробки (лёгкие параллельные запросы к БД и Redis)
— Pydantic для строгой валидации данных запроса
— Быстрая разработка (для e-commerce важны частые релизы)	Django: слишком тяжёлый, синхронный по умолчанию, неудобный под API-first подход
База данных	PostgreSQL 16	— ACID-транзакции для резервирования остатков товара
— Поддержка сложных JOIN (каталог ↔ цены ↔ остатки)
— Полнотекстовый поиск для ILIKE по артикулам	MySQL: слабее в сложных запросах и транзакциях с высоким конкурентным доступом
Кеш	Redis 7.2	— Хранение результатов поиска (структура ключ-словарь)
— In-memory = низкая задержка
— TTL для автоматического обновления кеша (10 минут)	Memcached: только key-value (неудобно для сложных результатов поиска с метаданными)
Балансировщик	Nginx	— Отдача статики (фронтенд)
— Проксирование API
— Простое масштабирование (добавление upstream-серверов)	Apache: хуже держит большое число соединений на статике
Фронтенд (веб)	React 18 + TypeScript	— Компонентная архитектура для каталога
— TypeScript ловит ошибки типов при обмене с API (цены, артикулы — числа/строки)
— Большое сообщество	Vue: команда больше знает React
Мобильное приложение	Flutter (для кросс-платформы)	— Один код под iOS и Android
— Быстрая разработка	React Native: Flutter сейчас стабильнее для сложных списков (каталог запчастей)

Детальное описание слоёв для каждого модуля
Модуль search (поиск запчастей)
text
SearchController
    ├─ GET /search?q={query}
    ├─ Валидация: query не пустой, длина > 2 символов
    └─ Вызов search_service.search(query, filters)
    
SearchService
    ├─ Проверка Redis: есть ли ключ `search:{hash_of_query}`
    │   └─ Если есть → вернуть из кеша (TTL 10 мин)
    ├─ Если нет → вызов search_repo.fulltext_search(query, filters)
    └─ Сохранить результат в Redis (набор ID запчастей)
    
SearchRepository
    └─ SELECT * FROM parts 
       WHERE (article ILIKE %s OR name ILIKE %s) 
         AND category = %s 
         AND price BETWEEN %s AND %s
Модуль orders (оформление заказа) — ключевой с транзакцией
text
OrderController
    ├─ POST /orders (auth required)
    ├─ Валидация: корзина не пуста, способ доставки выбран
    └─ Вызов order_service.create_order(user_id)

OrderService
    ├─ Получить корзину пользователя (из Redis или cart_repo)
    ├─ Вызвать stock_repo.reserve_items(items) ВНУТРИ ТРАНЗАКЦИИ
    └─ Создать заказ (order_repo.create)

OrderRepository + StockRepository (один сеанс БД)
    └─ with db.transaction():
        1. SELECT stock FROM parts WHERE id IN (...) FOR UPDATE
        2. UPDATE stock SET quantity = quantity - %s WHERE id = %s
        3. INSERT INTO orders (user_id, items, total) VALUES (...)
        4. DELETE FROM cart_items WHERE user_id = %s
Почему такая архитектура?
Слой	Что даёт
Controller	Единое место валидации, отделение формата запроса от логики
Service	Бизнес-правила (скидки, проверка остатков, расчёт доставки) — легко тестировать в юнит-тестах
Repository	Смена БД или добавление кеша без изменения бизнес-логики
Cache Layer	Прозрачное кеширование для модуля Search
Связи между модулями (зависимости)
text
Orders ──uses──► Stock      (резервирование)
Orders ──uses──► Cart       (очистка корзины после заказа)
Search ──uses──► Catalog    (поиск обращается к данным каталога)
Auth ──provides JWT ──► все модули, требующие аутентификации
Никаких циклических зависимостей — Orders вызывает Stock и Cart, но не наоборот.
